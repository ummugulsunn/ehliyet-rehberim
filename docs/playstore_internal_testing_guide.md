# Play Store Internal Testing Configuration Guide

## Overview

This guide provides step-by-step instructions for setting up and managing Google Play Store Internal Testing for the Ehliyet Rehberim app, specifically focusing on testing the Google Sign-In functionality fixes.

## Prerequisites

- [ ] Google Play Console access with app management permissions
- [ ] Release APK built and ready for upload
- [ ] Firebase Console configured with release SHA-1 fingerprints
- [ ] Test user email addresses collected

## Phase 1: Play Console Setup

### 1.1 Access Play Console

1. Go to [Google Play Console](https://play.google.com/console)
2. Sign in with your developer account
3. Select the "Ehliyet Rehberim" app from your app list

### 1.2 Navigate to Internal Testing

1. In the left sidebar, click **"Testing"**
2. Select **"Internal testing"**
3. You should see the internal testing track overview

### 1.3 Create New Release

1. Click **"Create new release"** button
2. You'll be taken to the release creation page

## Phase 2: APK Upload and Configuration

### 2.1 Upload Release APK

1. In the "App bundles and APKs" section:
   - Click **"Upload"**
   - Select your release APK file (generated by `prepare_playstore_release.sh`)
   - Wait for upload to complete

2. Verify APK details:
   - **Package name**: Should be `com.ehliyetrehberim.app`
   - **Version code**: Should match your build
   - **Target SDK**: Should be appropriate for current Android versions
   - **Permissions**: Review the permissions list

### 2.2 Add Release Notes

### 2.3 Review Release

1. Click **"Review release"**
2. Review all details:
   - APK information
   - Release notes
   - Target audience (Internal testing)
3. If everything looks correct, click **"Start rollout to Internal testing"**

## Phase 3: Test User Management

### 3.1 Add Test Users

1. Go to **"Testing"** > **"Internal testing"**
2. Click on **"Testers"** tab
3. Click **"Create email list"** or use existing list

#### Recommended Test Users (10-20 users):
- Development team members
- QA team members
- Product managers
- Selected beta users
- Device variety representatives

### 3.2 Configure Email List

1. **List name**: "Ehliyet Rehberim Internal Testers"
2. **Add emails**: Enter test user email addresses (one per line)
3. **Save** the email list

### 3.3 Generate Testing Link

1. After adding testers, you'll see a **"Copy link"** button
2. Copy the internal testing link
3. This link will be used in invitations to testers

## Phase 4: Test User Invitations

### 4.1 Send Invitations

Use the invitation template generated by the preparation script:

**Subject**: Ehliyet Rehberim - Internal Testing Daveti

**Body**: Use the content from `test_user_invitation_template.md`

### 4.2 Invitation Content Checklist

- [ ] Clear instructions on how to join testing
- [ ] Testing link included

- [ ] Contact information provided
- [ ] Testing timeline mentioned

## Phase 5: Monitoring and Management

### 5.1 Track Installation and Usage

1. Go to **"Testing"** > **"Internal testing"** > **"Statistics"**
2. Monitor:
   - Number of testers who joined
   - Installation rates
   - Crash reports
   - ANR (Application Not Responding) reports

### 5.2 Collect Feedback

#### Through Play Console:
- Monitor reviews and ratings in testing track
- Check crash reports and stack traces
- Review performance metrics

#### Through Direct Channels:
- Email feedback
- WhatsApp/Telegram groups
- Feedback forms
- Direct communication with testers

## Phase 6: Testing Scenarios Execution

### 6.1 Primary Test Scenarios

Ensure testers execute these critical scenarios:

1. **Google Sign-In Success Flow**
   - Fresh installation
   - Google account selection
   - Permission granting
   - Successful authentication

2. **Network Connectivity Issues**
   - No internet connection
   - Poor network conditions
   - Network recovery testing

3. **Fallback Authentication**
   - Google Sign-In failure simulation
   - Apple Sign-In testing (iOS)
   - Guest mode functionality

4. **Authentication State Management**
   - App restart with logged-in user
   - Sign-out functionality
   - State persistence testing

### 6.2 Device Coverage

Ensure testing across:
- **Android Versions**: 5.0, 6.0, 7.0, 8.0, 9.0, 10, 11, 12, 13
- **Manufacturers**: Samsung, Google, OnePlus, Xiaomi, Huawei
- **Screen Sizes**: Small, Normal, Large, Extra Large
- **RAM**: 2GB, 4GB, 6GB, 8GB+

## Phase 8: Issue Resolution Process

### 8.1 Issue Classification

**Critical Issues** (Fix immediately):
- App crashes on startup
- Google Sign-In completely broken
- Data loss or corruption
- Security vulnerabilities

**High Priority Issues** (Fix within 24-48 hours):
- Google Sign-In fails for specific devices
- Performance degradation
- UI/UX problems affecting core functionality

**Medium Priority Issues** (Fix within 1 week):
- Minor UI glitches
- Non-critical feature issues
- Localization problems

**Low Priority Issues** (Fix in next release):
- Enhancement requests
- Minor cosmetic issues
- Edge case scenarios

## Troubleshooting Common Issues

### Issue: Testers Can't Find the App

**Solution**:
1. Verify testers are using the correct Google account
2. Check if testers have joined the testing program
3. Ensure testing link is correct and not expired
4. Verify testers are in the correct country/region

### Issue: Google Sign-In Still Failing

**Solution**:
1. Verify release SHA-1 is added to Firebase Console
2. Check google-services.json is updated
3. Confirm APK is signed with correct keystore
4. Test with different Google accounts

### Issue: Low Tester Participation

**Solution**:
1. Send reminder emails
2. Provide clearer testing instructions
3. Offer incentives for participation
4. Simplify the testing process

### Issue: Inconsistent Test Results

**Solution**:
1. Provide more detailed test scenarios
2. Create video tutorials for testing
3. Standardize testing environment
4. Improve communication channels
